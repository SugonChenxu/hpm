## 0. 现状速览（部署相关，已核实）

| 项 | 现状 | 对内网网页+多用户的影响 |
|----|------|------|
| 架构 | Express 单进程托管 API + 前端静态文件 | ✅ 单份部署即可多人访问 |
| 监听 | `app.listen(3000)`，绑所有网卡 | ✅ 局域网同网段可访问 |
| 前端 API | `BASE = "/api"`（相对路径） | ✅ 局域网网页几乎免 host 改造 |
| 数据库 | SQLite（`server/hpm.db`，相对路径） | 🟡 单写锁；内网小团队 OK，高并发需评估 |
| 鉴权 | **无** | ⚠️ 必须新增登录认证 |
| 数据模型 | 所有表**无 `user_id`/`owner_id`** | ⚠️ 必须全表加 owner 隔离 |
| 原生依赖 | `better-sqlite3@11`（预编译） | ✅ 服务器一份即可，随部署包附 |
| 构建依赖 | `vite@8`（Node ≥ 20.19/22） | ✅ 服务器内置 Node 22 |
| Mantis | 路由齐全，存 `mantis_connection` 单行 | 🟡 改为 per-user（加 owner_id） |
| OA 导入 | 书签向 `localhost:3000` 注入 | 🟡 需带当前用户 token，否则分不清归属 |
| 腾讯会议/PLM | 适配器存在，绑个人 OAuth | ❌ 不纳入推广 |

## 1. 部署形态（已决策：内网网页集中部署）

### 决策结论
- **集中部署**：一台**常开机器**（你的办公电脑常开 / 一台内网服务器）跑一份 Forge，同事浏览器访问 `http://<服务器IP>:3000`，**零安装**。
- **各自独立管理**：每个用户注册独立账号，数据按 `owner_id` 隔离，互不可见、互不干扰。无角色权限（人人平等）。
- **同一局域网**：免公网 HTTPS/域名，复杂度可控。
- **外部对接**：Mantis 必须（每用户各自配 Token）；OA 选装；腾讯会议/PLM 不做。

### 与原方案A（每人本机绿色包）的区别
- 方案A = 每人本机一份 → 数据天然隔离，但每人需维护一份。
- 本方案 = 一份集中 + 多用户隔离 → 同事零维护，但需做登录+隔离改造。

### 部署包定位
- 原 `forge-package/`（绿色包）转为**「服务器部署包」**：解压到那台常开机器，用 PM2/服务常驻即可；同事**不需要此包**，只需浏览器+网址。
- 同事侧体验：收到网址 → 浏览器打开 → 注册/登录 → 即用。

## 2. 多用户改造 —— 功能断点清单

> 这是本次从"单用户"转向"多用户"必须补的工程。

1. **登录认证（无 → 必须有）**
   - 新增 `users` 表（username, password_hash, created_at）
   - 注册 / 登录 / 登出接口（bcrypt 哈希 + Session 或 JWT）
   - 认证中间件：保护所有 `/api/*`，从 token 解析 `userId`
2. **数据隔离（全表加 `owner_id`）**
   - 受影响表：projects, phases, tasks, issues, materials, meetings, meeting_action_items, weekly_reports, week_meetings, week_meeting_outputs, schedule_tasks, mantis_connection, phase_templates（新用户初始化自己的模板）
   - 所有 SELECT/INSERT/UPDATE/DELETE 加 `WHERE owner_id = :userId`
3. **种子数据 per-user**
   - 新用户注册时，初始化一套「曙光标准流程」默认阶段模板（原全局种子改为 per-user）
4. **前端登录态**
   - 登录页、token 存储、API client 加 `Authorization`、路由守卫、登出按钮、当前用户展示
5. **Mantis per-user**
   - `mantis_connection` 加 `owner_id`，每人配自己的地址+Token
6. **OA 导入带 token**
   - OA 浮窗脚本需把当前用户 token 一并 POST，否则物料归不到正确 owner
7. **SQLite 并发**
   - 多人同写：better-sqlite3 同步 + WAL 模式，内网小团队基本够用；明确"高并发写需换 Postgres"的边界

## 6. 账号管理与迁移实操（已落地 · 2026-07-20）

> 阶段 0–2（认证基础设施 + 路由 owner 隔离 + 前端登录）已**全部完成并验证通过**。

### 6.1 本人首次登录
- 初始管理员账号：`chenxu`
- 初始密码：**（由迁移脚本随机生成，见下方「查看/重置密码」）**
- 登录地址：`http://<服务器IP>:3000/` → 输入用户名/密码 → 进入。
- 本人全部历史数据（6 个项目 / 任务 / 故障 / 物料 / 会议纪要等）已自动绑定到 `chenxu`，登录后即可见，**不会空白**。

### 6.2 给同事开账号（手动逐个添加）
脚本：`server/scripts/create-user.js`（依赖 `bcryptjs`，在 `server/` 目录运行）。

```bash
cd D:/HPM/server
# 方式一：参数式（推荐，一行建好）
node scripts/create-user.js 同事用户名 同事密码

# 方式二：交互式（逐个提示输入）
node scripts/create-user.js

# 方式三：已存在则仅重置密码
node scripts/create-user.js 同事用户名 新密码 --reset
```
- 密码规则：明文不入库，bcrypt 哈希存储；长度 ≥ 6 位。
- **不开放注册**：同事不会看到"注册入口"，只能等你建好账号后登录。

### 6.3 查看 / 重置本人或同事密码
- 直接重跑 `create-user.js <用户名> <新密码> --reset` 即可覆盖。
- 若忘记 `chenxu` 初始随机密码，用同样命令重置：
  `node scripts/create-user.js chenxu 你想设的密码 --reset`

### 6.4 数据隔离（已验证）
- 每个用户只能看到自己 `owner_id` 下的数据；未登录访问任意 `/api/*` 返回 `401`。
- 验证记录：同机双账号测试用例通过（A 用户看不到 B 用户的项目/任务）。

### 6.5 迁移到公司内网服务器（演进路径）
- 停止服务 → 拷贝 `server/hpm.db`（单文件，含 `users` 表）→ 放到目标机同样路径 → 启动即可。
- 代码无需改动；`users` 与业务数据随库一起迁移，账号与归属关系完整保留。
- 注意：会话目录 `server/sessions/` 与备份目录 `server/backups/` 已被 `.gitignore` 忽略，不随仓库走，迁移时按需另行处理。

## 已决策项（2026-07-20 · 含细化）
1. 方向：**内网网页集中部署**（一台常开机器，同事浏览器访问，零安装）。
2. 数据：**各自独立管理**（登录 + 全表 owner_id 隔离，无角色权限）。
3. 网络：**同一局域网**（免 HTTPS/域名）。
4. 外部对接：**Mantis 必须**（per-user）；**OA 选装**；**腾讯会议 / PLM 不做**。
5. 推翻：12:19 定的「方案A 绿色包每人本机」。
6. 认证方式：**Session**（服务器会话，内网小团队简单够用；非 JWT）。
7. 账号开通：**你手动逐个添加**（不开放注册；提供建账号脚本，每个同事单独建）。
8. 服务器机器：**就用你本机办公电脑常开**（不另购/另找独立服务器；数据先放本机）。
9. 会话时效：**尽量长**（30 天或持久「记住我」，减少同事重复登录）。
10. 现有数据迁移：先为本人建专属账号（chenxu），将现有全量数据绑定到该 owner_id，避免登录后空白。
11. 演进路径：**先本机试用 → 后整包迁移公司内网服务器**（SQLite 单文件拷贝即迁移，代码无需改；见 §6）。
