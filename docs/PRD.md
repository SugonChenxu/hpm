## 一、产品目标

### 1.1 一句话定位
面向硬件项目经理的**个人级全生命周期项目管控工具**，覆盖从预研立项到新品导入（NPI）的全部研发阶段，以曙光硬件产品开发流程为默认模板，支持自定义阶段与任务。

### 1.2 核心价值
| 维度 | 当前痛点 | HPM 解决方式 |
|------|---------|-------------|
| **进度掌控** | 多项目并行时难以快速感知各项目所处阶段与风险 | 看板+甘特图双视图，阶段门禁自动计算转段准入状态 |
| **文档管理** | PRD/评审报告/计划书分散在 PLM/邮件/本地 | 以项目为容器，集中关联所有交付物及其版本 |
| **需求追踪** | 需求→设计→测试追溯链断裂 | 需求池 → 模块映射 → 测试用例关联 → 达成率看板 |
| **物料跟进** | 备料交期靠邮件和 Excel，漏单缺料频发 | 物料清单 + 备料周期预警 + 交期状态跟踪 |
| **缺陷管理** | Mantis 与项目进度割裂，DI 值靠人工统计 | 内置缺陷登记 + 后续对接 Mantis API 自动同步 |
| **标准化** | 各项目经理流程执行标准不一 | 预设曙光标准流程模板(M1-M5/L1-L12)+ 支持自定义裁剪 |

### 1.3 非目标（明确不做）
- ✗ 多用户协作与权限体系（个人工具，MVP 仅单用户）
- ✗ 正式文档导出（不做 Word/PDF 模板生成）
- ✗ 对接 SAP/OA/ERP（仅预留 Mantis 和 PLM 接口）
- ✗ PWA 离线模式（仅预留响应式 + Service Worker 接口）
- ✗ 实时通知/消息推送（不做钉钉/飞书/企微集成）

## 三、需求池

### 3.1 P0 — MVP 核心（第一版必须交付）

| 编号 | 模块 | 需求描述 | 对应故事 |
|------|------|---------|---------|
| F-01 | **项目管理** | 创建/编辑/删除项目，基本信息：代号、名称、类别（新品/OEM/升级/定制/派生）、状态 | US-01 |
| F-02 | **流程模板** | 预设「曙光硬件产品开发标准流程」模板（M1-M5 五阶段 + L1-L12 任务 + TR/DCP/MR 评审点 + 转段 DI 阈值），可一键应用到新项目 | US-01 |
| F-03 | **阶段管理** | 自定义阶段：对项目增/删/改名/调整阶段顺序；每个阶段可定义名称、目标描述、计划工期 | US-03 |
| F-04 | **里程碑/门禁** | 每个阶段可添加入口/出口条件（门禁点名称、类型 TR/DCP/MR/G-O、DI 值门槛）；系统根据条件状态自动判定是否满足转段 | US-05 |
| F-05 | **任务管理** | 每个阶段下添加任务，含：任务名、责任岗位、计划开始/结束日期、状态（未开始/进行中/已完成） | US-04 |
| F-06 | **项目看板** | 首页仪表盘：所有项目卡片列表，显示代号/名称/当前阶段/进度百分比/风险等级色标；支持按状态/类别筛选 | US-02 |
| F-07 | **需求管理** | 需求池 CRUD：需求编号、标题、所属模块、优先级(P0/P1/P2)、状态、关联项目 | US-06 |
| F-08 | **物料清单** | 物料 CRUD：料号、名称、规格、数量、供应商、计划交期、实际到货日期、状态（待下单/已下单/在途/已到货） | US-08 |
| F-09 | **项目详情页** | 项目详情：基本信息区 + 阶段时间线(横向可视化) + 当前阶段任务列表 + 快速切换阶段视图 | US-02 |

## 五、数据模型（概要）

基于知识库八类核心实体，初步划定数据表：

| 表 | 核心字段 | 关系 |
|----|---------|------|
| `projects` | id, code, name, category, status, template_id, created_at | 1:N → phases |
| `phase_templates` | id, name, description, is_preset（曙光标准=预设不可删） | 1:N → phase_template_items |
| `phase_template_items` | id, template_id, order, name, type(PHASE/GATE), duration_weeks, di_threshold | — |
| `phases` (项目实例) | id, project_id, template_ref_id, name, order, planned_start, planned_end, actual_start, actual_end, status | 1:N → tasks, gates |
| `gates` | id, phase_id, name, type(TR/DCP/MR), di_threshold, is_passed, passed_at | — |
| `tasks` | id, phase_id, name, role, planned_start, planned_end, status | — |
| `requirements` | id, project_id, code, title, module, priority(P0/P1/P2), status, is_met | 1:N → test_cases（P1） |
| `materials` | id, project_id, part_no, name, spec, quantity, supplier, lead_time_days, planned_delivery, actual_delivery, status | — |
| `issues` | id, project_id, code, severity, title, phase_id, requirement_id, status, di_weight(P1) | — |
| `documents` | id, project_id, phase_id, name, version, file_path, author, review_status(P1) | — |

## 六、技术约束与架构约定

| 约束项 | 决策 |
|-------|------|
| 技术栈 | 前端 Vite + React + MUI + Tailwind CSS；后端 Node.js + Express/Fastify（待架构师确认）；数据库 SQLite（个人用） |
| 架构模式 | 前后端分离，RESTful API |
| 用户系统 | 个人使用，MVP 无登录/无多用户；后端预留 Auth 中间件接口 |
| 阶段模板 | 数据库存储模板定义，项目创建时从模板实例化为项目阶段（非硬编码） |
| 外部对接 | Mantis/PLM 对接接口预留（abstract service layer + adapter pattern），MVP 不实现 |
| 移动端 | 前端使用响应式布局（MUI Grid/Breakpoints），预留 PWA Service Worker 注册点 |
| 版本管理 | 每次 git push 自动生成 CHANGELOG；代码地址 https://github.com/SugonChenxu/hpm.git |
| 部署 | 待定（候选：Vercel + 自部署） |

## 七、UI 设计要点

### 7.1 设计语言
- **配色**：专业硬件工业风——深蓝主色(#1565C0) + 灰白背景 + 红黄绿状态色
- **字体**：系统默认中文字体（苹方/Microsoft YaHei）
- **组件**：MUI 5 组件库 + Tailwind 辅助样式

### 7.2 关键页面线框

**仪表盘**：顶部项目统计卡片（总数/进行中/高风险/逾期物料数）→ 下方项目卡片网格（每张卡片：代号大标题 + 阶段进度条 + 风险色标 + 最后更新）

**项目详情**：左侧阶段时间线纵向（M1→M5 类似地铁线路图，当前阶段高亮）→ 右侧内容区根据选中阶段切换（任务列表/需求/物料/缺陷/文档）

**阶段时间线**：横向或纵向可视化，类似 Jira 的 Roadmap，每个节点显示阶段名+工期标签+状态色
