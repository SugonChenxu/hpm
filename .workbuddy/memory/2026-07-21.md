## M8 权限模型升级为 owner/admin/member 三级（已实装部署 + 推送 commit a54681c）
- 用户指出原"人人可管"设计不合理，采纳方案 B(role 字段)+②(bootstrap 指定主账号为 admin)+③(owner/admin/member 三级) 组合。
- **数据库** `server/src/db.js`：`users` 表 ALTER 加 `role` 列(`TEXT NOT NULL DEFAULT 'member'`)；迁移时把 `chenxu` 升为 owner（不可撼动的最高权限）。
- **后端**：`auth.js` login/me 返回 role；`routes/users.js` 新增 `requireAdmin`(owner/admin)/`requireOwner`(owner) 中间件，`/me/password` 所有人可用、其余套 requireAdmin，新增 `PUT /api/users/:id/role`（仅 owner，禁改 owner 角色、禁设为 owner）。
- **权限矩阵**：owner 不可被删/降级/重置密码；admin 可管 member 但不能碰其他 admin/owner；member 仅改自己密码、进不了用户管理页（侧边栏隐藏 + 入口守卫）。
- **前端**：`Sidebar.jsx` 仅 owner/admin 显示用户管理菜单；`UserManagementPage.jsx` 入口守卫 + 角色列(Chip) + 按角色禁用重置/删除 + owner 专属「修改角色」对话框；`api/client.js` 加 setRole。
- **验证**：端到端权限矩阵脚本 25/25 全通过（含越权被拒 403/400）；脚本临时接管并恢复 chenxu 密码、清理测试账号，零污染。
- **文档**：重写 08-PRD 为三级模型 + CHANGELOG 补条目 + 顶层 PRD M8 描述与修订记录更新。

### 经验沉淀（可复用）
- 最小权限原则落地：用 `role` 列 + 中间件(requireAdmin/requireOwner) 比"固定 admin 账号"更灵活（管理员可多个、不与账号名死绑）。
- 零污染端到端验证模式：脚本临时改管理员密码 → try{跑测试} → finally{恢复原密码 + 删测试数据}，避免碰真实账号。

## 缺陷排查：OA 浏览器插件导入物料「项目错配」（已修复 + 推送）
- **现象**：用户反馈"从浏览器插件 OA 导入【物料管理】不好使"。
- **根因**：M8 多用户隔离改造（767328c）给所有 `/api` 加 `requireAuth`，但 `auth-middleware.js` 白名单只放行了 `/materials/oa-import`、`/materials/oa-fetch`，**漏掉插件匹配项目用的 `GET /api/projects`** → 插件匿名调用被 401 → 拿不到项目列表 → 所有 OA 物料被塞进写死的 project 20（液冷超节点）。注：`oa-import` 接口本身正常（白名单内，返回 200 能落库），所以是"项目错配"而非"完全失败"。
- **修复**：
  1. 后端 `POST /api/materials/oa-import` 新增按 `internal_code`（内部立项号）/ `order_number` **服务端解析目标项目**，不再依赖插件调 `/api/projects`；解析优先级 internal_code → project_id → 报错；响应回显 `project_id`/`project_name`。
  2. 插件 `chrome-extension/inject.js` 移除 `/api/projects` 调用，改为上报已提取的「内部立项号」为 `internal_code`；成功提示显示项目名称。
  3. 用本机 Chrome 重新打包 `chrome-extension.crx`（chrome-extension.pem）。
- **验证**：curl 三种场景全对（按 701085 → project 6 HG9400整机柜；凭空立项号 → 报错；无立项号 → 回退 20）。
- **工具链经验**：可在 Git Bash 用 `"/c/Program Files/Google/Chrome/Application/chrome.exe" --headless=new --user-data-dir=临时目录 --pack-extension=扩展目录 --pack-extension-key=.pem` 重新打包 .crx（无 GUI，自动覆盖原 .crx）。
- **安全说明**：`oa-import` 仍属白名单公开接口（与 767328c 前一致），LAN 内知道立项号者可 POST；如需加固可后续加 OA 专用 token。

### Mantis Cookie 取用指南（给非 Chrome 同事）
- 结论：Mantis Cookie 取自**同事自己登录 Mantis 的浏览器**，与打开 Forge 用哪个浏览器无关 → Edge/Firefox/Safari 均可，无需 Chrome。
- 关键坑：Mantis 登录态 Cookie 通常是 `httpOnly`，JS 书签/`document.cookie` 读不到 → 必须走 DevTools → Network → 复制请求 Headers 里的 `Cookie` 整段值（已写 `docs/mantis-cookie-guide.md` 可转发）。
- 备选：右键请求「Copy as cURL」→ 取 `--cookie '...'` 段。
- Cookie 会随 Mantis 会话过期，过期后 Forge 报「鉴权失败」，需重取。
- **已内嵌进界面**：`MantisConnectionCard.jsx` 在 Cookie 框 helperText 加「？如何获取 Cookie」链接 → 弹出 Dialog（含各浏览器通用步骤 + httpOnly 关键坑 + cURL 备选）。前端已构建部署，无需另开文档。修复了 return 双根节点需 Fragment 包裹的 JSX 报错。
