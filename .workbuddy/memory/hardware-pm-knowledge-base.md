## 一、核心业务域与关键对象

| 实体 | 说明 | 关键属性 |
|------|------|---------|
| 项目(Project) | 一个完整的硬件产品开发任务 | 代号/类别/状态/当前阶段 |
| 阶段(Phase) | M1–M5 大阶段，内含 L1–L12 任务级 | 编号/名称/开始结束条件/工期 |
| 里程碑(Milestone) | 阶段间决策/评审节点(TR/DCP/MR/G-O) | 评审名/判定结果/DI值 |
| 文档(Document) | 各阶段正式产出物 | 名称/版本号/责任人/评审状态 |
| 角色(Role) | 项目参与人员职能分类 | 姓名/岗位/职责/部门 |
| 需求(Requirement) | 产品必须满足的功能/非功能规格 | 编号/模块/优先级/达成状态 |
| 物料(BOM) | 产品零部件清单(通用+开发) | 料号/名称/交期/供应商/SS状态 |
| 缺陷(Issue) | 测试/生产发现异常 | 编号/严重度/状态/DI贡献 |

**关键缩写**：M1预研/M2计划/M3研发测试/M4试制/M5新品导入,NPI；L1-L12任务层级；TR1-TR5技术评审；DCP1立项/DCP2量产决策；MR1-MR3管理评审；EVT工程验证(标准9周)；DVT设计验证(标准12周)；PVT生产验证(~6周)；G/O板卡版本发布评审；DI缺陷指数；DQA设计质量保证；TPM测试代表；SIT系统集成/SI信号完整性/DC电源/PA产品保证。

## 二、研发阶段 / 流程全景

### M1 预研 → DCP1  
- **L1 预研**：市场代表主导，行业BD/SE全局规划，识别技术风险
- **L2 详细需求**：产品经理主导，编制PRD + 典配配置 + 通用物料
  - TR1 产品需求评审(DQA组织)：评审PRD+典配，由直属领导/总工/市场/制造/服务/架构师有驳回权
- **L3 立项筹备**：PM主导立项建议书/报告/估算表/检查表
  - TR2 系统总体方案评审(DQA)
  - DCP1 立项决策评审 → M1出口

### M2 计划 → MR1
- **L4 概要设计**：架构师主导，DCP1后启动，依据PRD
  - 5项输出：概要设说明书(含六性设计)+散热仿真+测试配置+发布策略+Early BOM
  - TR3 概要设计评审
- **L5 开发计划**：PM主导，输出项目计划书+物料计划+可行性估算+BIOS/BMC版本计划+质量策划(DQA)
  - MR1 计划评审 → M2出口

### M3 研发测试 → MR2/MR3
- **L6 详细设计**：TR2/TR3完成后启动，多角色并行
  - 板卡：原理图→叠层(4天)→Layout→SI仿真→Gerber out(硬件/电源/SI/Layout)
  - 结构：机箱/刀箱/导风罩/风扇仓/Cage/包装(结构/包装工程师, 备料4-6W)
  - 散热：风冷/液冷散热器+风扇(散热工程师, 备料4-6W)
  - FW：BIOS/BMC/CMM 详细设计
  - ID：前面板/箱耳/丝印/标签/贴纸
  - 线缆：系统架构师/硬件工程师确认规格，线缆工程师跟进
  - 同步制定 Power-On 计划 + EVT 测试计划
  - 备料关键数据：主板备料2月/小板卡1月/PCB mid loss 2-3周/low loss 3-4周
  - TR3A 详细设计评审
- **L7 Power-On (2W)**：HW/DC/BIOS/BMC/CMM/System 多专业并行开机验证
- **L8 EVT (9W)**：Engineering Verification Test——尽可能发现设计问题
  - DC(4W)/SI(4-6W:无源1W+有源)/SIT(前4W重点)/散热(~6W)/可靠性+PA
  - 软件：BMC/BIOS/CMM 两周一迭代
  - 机构安装评审 → TR4原型机技术状态评审 → B01 G/O
  - MR2 DVT准入评审(DI<130) → 进入DVT
- **L9 DVT (12W)**：Design Verification Test——整机功能完整性+产品定型
  - 全功能测试+Second Source验证+线缆开号
  - 机构开模：NCT样品→报价2-3W→招标→试模→生产
  - TR4A工程样机技术状态评审 → C01 G/O(主板定版)
  - MR3 PVT准入评审(DI<30) → 进入M4

### M4 试制 → 质量评审
- **L10 首件/批量试制**：11项准备工作(部件规格书→料号→BOM→首批采购→生产技术要求→排产→入库)→工艺评审→试制状态检查→首件检验→首件鉴定→批量试制(含老化)
- **L11 PVT (~6W)**：SIT+可靠性测试 → TR5初始产品技术状态评审
- **L12 批量测试**：使用产线机器进行批量可靠性测试 → DQA质量评审(DI≈0-1) → 软件版本定版 → NPI准入
- **裁剪矩阵**：M强制/R建议/无标识可裁剪；新品开发全M；OEM整机全M；定制产品全M

### M5 新品导入 → 结项
- NPI验收(A/B整机类：质量评审→NPI；C类：仅质量报告)
- 资产清算(库位物料退库/移库/领用)→费用确认→关闭订单号
- 结项决策评审(PMO组织, 领导层+品质+产品经理+采购参加)
  - 核对结项检查表 → 结项评审报告
- 系统结项申请(OA/PLM: 上传立项+结项+资产+费用报告; 预算>1500万需审计)
- 项目文档归档(PLM更新+组织过程资产更新)

## 三、关键技术要点

### 3.1 六性设计(GJB参考)
| 六性 | 设计关注点 | 责任人 |
|------|-----------|--------|
| 可靠性 | MTBF>10万h(25℃/80%负载)/模组化设计 | 产品经理B |
| 维修性 | 免工具拆卸/组件独立装卸 | — |
| 测试性 | 开发功能均可测试 | — |
| 保障性 | 参考曙光硬件设计规范 | — |
| 安全性 | 安规EMC(≤1350W)/温度/湿度/海拔/噪声 | — |
| 环境适应性 | 高低温存储/工作/高温高湿 | — |

### 3.2 三大主线(立项报告结构)
1. **商业计划** → 投资管理团队
2. **需求实现** → 产品开发团队
3. **项目管理** → 项目管理团队

### 3.3 DI值转段标准
| 转段 | DI门槛 | 说明 |
|------|--------|------|
| EVT→DVT | <130 | DQA统计,BUG权重化累计 |
| DVT→PVT | <30 | 需达到标准方可参加MR3 |
| PVT→量产 | <10(一般1或0) | 质量评审门槛 |

### 3.4 关键件与重要件
- **关键件**：含关键特性，不满足将危及人身安全并导致不能完成主要任务
- **重要件**：不含关键特性但含重要特性，不满足将导致不能完成主要任务

### 3.5 立项建议三条建议(来自行业SE经验)
1. 需求完备性：前期通盘考量避免范围蔓延
2. 机型优先级评估+资源考量错峰：避免多任务并发风险
3. 架构师提前识别技术风险并规划预研方案：避免详设阶段更改方案

## 五、质量门禁与评审点

```
M1 → TR1+TR2 → DCP1 → M2 → TR3+MR1 → M3 →
  L6/L7 → TR3A → Power-On → L8(EVT) → TR4+B01G/O+MR2(DI<130) →
  L9(DVT) → TR4A+C01G/O+MR3(DI<30) →
M4 → L10(试制+工艺+首件+批量) → L11(PVT)+TR5 → L12(批量) → 质量评审(DI≈0-1) →
M5 → NPI验收 → 结项决策评审 → 系统结项 → 文档归档
```

关键门禁不通过处置：
- TR不通过→问题闭环后重新评审
- DCP不通过→不予立项/不予转段
- MR不通过(DI不达标)→不得参加下一阶段
- 质量评审不通过→整改后重新验证
- 结项不通过→解决评审问题再次评审

## 七、可借鉴的全栈开发工作流方法论（来自 9 张 Codex 工作流图）

### 7.1 方法论总览(来源：同济博一·Codex全栈工作流)

这 9 张图完整展示了一种"AI 驱动全栈开发"的工作流范式，核心理念是：**想法→AI改代码→GitHub同步→Vercel部署→域名实时预览**。

#### 八步工作流(Step 1-8)：

**Step 1 — VS Code 配置 Codex**：在 VS Code 中安装 OpenAI Codex 插件，配置模型和 API，创建项目目录环境。

**Step 2 — Git & GitHub 配置**：建立版本控制体系作为安全网——每条更改都有 Git 记录，可随时回滚复原。

**Step 3 — 给 Codex 配置 Skills(技能)**：将自定义规则/工作流/SOP 配置为可在对话中调用的 Skill，建立项目专属的"可复用能力库"。核心 Skills 包括：
- **GitHub Skill**：自动化 Git 操作与 GitHub 交互
- **Figma Skill**：设计稿→代码的转换
- **飞书 Skill**：团队协作与通知集成
- **前端设计与图文排版 Skill**：页面美学与排版规则
- **视觉与体验优化 Skill**：交互细节
- **产品研究与项目规则 Skill**：项目需求与业务规则沉淀

**Step 4 — 获取 AI API**：配置 Open AI API Key，赋予 Codex 访问大模型的能力。

**Step 5 — Vercel 部署上线**：前端项目通过 Vercel 自动部署，Git push 触发自动构建，生成公网可访问链接。

**Step 6 — Supabase 搭建数据库**：使用 Supabase(PostgreSQL)作为后端数据库服务，先设计表结构，再生成代码。提供原生身份认证(Auth)、实时订阅(Realtime)、文件存储(Storage)。

**Step 7 — 购买域名 & 解析到 Vercel**：阿里云等平台购买自定义域名，配置 DNS 解析到 Vercel 的 IP，让普通用户可访问。

**Step 8 — 手机端远程控制 Codex**：通过企业微信/飞书等即时通讯工具远程向 Codex 发送任务请求，实现移动端迭代闭环。

### 7.2 核心迭代闭环

```
[想法/需求]
    ↓
[手机(企业微信)提需求] ──→ [Codex 在 VS Code 接收并改代码]
    ↓                              ↓
[域名实时预览] ←── [Vercel 自动部署] ←── [GitHub Push 同步]
    ↑                              ↓
    └────── [Supabase 持久化数据] ←──┘
```

### 7.3 对本项目的启示

1. **环境先行原则**：Step 1-4 搭建"AI 工具链"比直接写代码更重要。对应本项目：先建 Vite+React 脚手架、配置 TailwindCSS/MUI、搭建 Node.js 后端框架、初始化数据库。

2. **版本控制即安全网**：所有代码修改 → git commit → push。用户约束中已明确要求"每次代码更新完强制 GitHub 同步"，与该方法论的"Step 2 是核心安全网"完全吻合。

3. **Skills = 可复用的工程知识**：用户要求"memory 初始化→知识基准库"本质上就是项目的第一个 Skill。后续每个模块的领域规则（评审体系/转段DI值/物料交期/角色职责）都可沉淀为 Skill 供开发时快速引用。

4. **部署流水线自动化**：Vercel 的 git-push-auto-deploy 理念可引入本项目——配置 GitHub Actions 实现前端/Vercel 自动部署 + 后端部署。

5. **数据库设计先行**：Step 6 "先设计数据库,再生成代码"对应本项目中应先完成 ER 图/表结构设计(项目/阶段/里程碑/文档/角色/需求/物料/缺陷 八张核心表)，再写后端 API。

6. **移动端访问 + 即时通讯集成**：用户作为硬件项目经理，需要手机查看项目状态/审批/进度。本项目应支持 PWA 移动端适配，并可考虑飞书/企业微信 Bot 集成实现状态推送。

## 八、待确认 / 冲突点

### 8.1 文档间不一致
- **DCP2 决策点**：仅在一处提及"DCP2"(量产决策)，大部分文档末提到，需确认是否有正式的量产决策评审环节。
- **EVT→DVT DI 值**：DVT 流程文档提及 DI<130，但浪潮流程(PPTX 10 Slide 3)给出的 DI 值不同(EPR→DVT<130是完全一致的)。需确认曙光内部标准版本。
- **Power-On 周期**：L4-L7 文档标 2W，但实际某些项目的 Power-On 启动物料到货时间可能不统一。

### 8.2 信息缺失
- **M1 预研阶段标准工期**：未找到 M1 明确的标准工期(不同项目差异大)。
- **GJB 六性设计的审计/合规要求**：仅提及 GJB 文档类型(标准化大纲/特性分析报告/标准化审查报告)，但未展开各文档的详细检查标准。
- **Mantis → PLM 统一系统迁移状态**：bug 管理仍用 Mantis，文档/流程管理用 PLM，存在两系统割裂。

### 8.3 需向用户澄清
1. 本 Web 项目管理工具的目标用户是**个人项目经理**还是**团队协作**？如果是团队，是否需要角色权限体系？
2. 是否需要对接现有的企业系统(PLM/Mantis/SAP/OA)？还是仅作为独立管理工具？
3. 项目进度跟踪是走**曙光流程的严格阶段制(M1-M5+L1-L12)**，还是支持**自定义阶段/里程碑**？
